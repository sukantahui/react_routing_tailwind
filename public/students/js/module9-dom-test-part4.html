Q1. What does event.preventDefault() do in a form submit handler?
  • A) Stops all JavaScript on the page
  • B) Prevents the browser‚Äôs default action for that event
  • C) Removes the event listener permanently
  • D) Stops the event from bubbling to parent elements

Correct Option: B

event.preventDefault() stops the browser's default behaviour for that event. For a submit event, it prevents the form submission (no page reload / navigation) so that you can handle validation or AJAX yourself.

Note: Option D is stopPropagation(), not preventDefault().

Q2. Output of the following click handler?
<button id="btn">Click</button>

<script>
  const btn = document.getElementById("btn");

  btn.addEventListener("click", function (event) {
    console.log(event.target.id);
    console.log(this.id);
  });
</script>
  • A) Logs: undefined and undefined
  • B) Logs: "btn" and undefined
  • C) Logs: "btn" and "btn"
  • D) Throws a runtime error

Correct Option: C

Here, the handler is a normal function, not an arrow function. In event listeners, this refers to the element on which the listener is registered (here, the button with id btn).

Also, event.target is the element that actually triggered the event, which is also the button. So both log "btn".

Q3. Which method stops the event from bubbling up the DOM tree?
  • A) event.stop()
  • B) event.stopImmediatePropagation()
  • C) event.stopPropagation()
  • D) event.preventDefault()

Correct Option: C

event.stopPropagation() stops the event from travelling further in the bubbling (or capturing) phase. stopImmediatePropagation() also stops bubbling BUT additionally prevents any other listeners on the same element from firing.

preventDefault() only blocks the default browser behaviour (like following a link).

Q4. What will be logged?
<ul id="menu">
  <li>Home</li>
  <li>Courses</li>
  <li>Contact</li>
</ul>

<script>
  const menu = document.getElementById("menu");

  menu.addEventListener("click", function (e) {
    console.log("You clicked:", e.target.tagName);
  });
</script>
  • A) Always logs "UL"
  • B) Always logs "LI"
  • C) Logs "LI" when a list item is clicked
  • D) Logs nothing

Correct Option: C

e.target is the actual element clicked. When clicking on a list item, the target will be that <li>, so tagName is "LI". This is a classic example of event delegation: the listener is on ul, but handles clicks from its children.

Q5. Which statement about event bubbling and capturing is TRUE?
  • A) Bubbling goes from parent to child; capturing goes from child to parent
  • B) Both bubbling and capturing go from child to parent
  • C) Capturing goes from parent to child; bubbling goes from child to parent
  • D) JavaScript no longer supports event capturing

Correct Option: C

In the capturing phase, events travel from the root (document) down to the target element. In the bubbling phase, they travel upwards from the target back to the root.

Browsers support both, though by default addEventListener listens in the bubbling phase.

Q6. What will be logged?
<button id="btn">Click</button>

<script>
  const btn = document.getElementById("btn");

  btn.addEventListener("click", () => {
    console.log("First listener");
  });

  btn.addEventListener("click", (e) => {
    e.stopImmediatePropagation();
    console.log("Second listener");
  });

  btn.addEventListener("click", () => {
    console.log("Third listener");
  });
</script>
  • A) First, Second, Third
  • B) Second only
  • C) First and Second only
  • D) Second and Third only

Correct Option: B

Event listeners run in the order they were attached. So: First ‚Üí Second ‚Üí Third. But in the second listener, e.stopImmediatePropagation() is called, which:

  • Stops any further listeners on the same element from running
  • Also stops bubbling

So only ‚ÄúFirst listener‚Äù and ‚ÄúSecond listener‚Äù? No ‚Äî when Second runs, it stops the Third, but the First already ran. Careful: in this code, stopImmediatePropagation is in second listener. Execution order: First ‚Üí Second ‚Üí STOP. Third does not run. So printed: First listener and Second listener. ‚úÖ Correct answer is actually C (First and Second only).

Q7. Which is the correct way to listen for a form submit event?
<form id="login-form">...</form>
  • A) document.addEventListener("submit", handler);
  • B) loginForm.onsubmit(handler);
  • C) loginForm.addEventListener("submit", handler);
  • D) addEventListener("submit", loginForm);

Correct Option: C

After selecting the form element (const loginForm = document.getElementById("login-form");), you attach the listener via loginForm.addEventListener("submit", handler).

Option A can work with capturing/ bubbling but is not the cleanest for one specific form. Option B is syntactically wrong ‚Äî onsubmit is a property, not a function. Option D has parameters reversed.

Q8. Output?
<input id="name" type="text" value="Coder">

<script>
  const input = document.getElementById("name");

  input.addEventListener("input", () => {
    console.log(input.value);
  });
</script>
  • A) Logs the new value every time the user types or deletes
  • B) Logs only once when page loads
  • C) Logs only when the input loses focus
  • D) Logs only when the user presses Enter

Correct Option: A

The input event fires on every change in the input‚Äôs value (typing, deleting, pasting), so it logs the updated value each time.

change would wait until the field loses focus.

Q9. Keyboard event: which property gives the key pressed?
  • A) event.code
  • B) event.key
  • C) event.whichKey
  • D) event.keyName

Correct Option: B

event.key returns the actual key value, like "a", "Enter", "Escape". event.code is more physical-key based (e.g., "KeyA").

Q10. Output of the following?
<form id="f">
  <input id="username" name="username">
  <button type="submit">Submit</button>
</form>

<script>
  const form = document.getElementById("f");

  form.addEventListener("submit", function (e) {
    console.log("Submitted");
  });
</script>
  • A) Logs "Submitted" and stays on the same page
  • B) Logs "Submitted" and then reloads / navigates
  • C) Does nothing
  • D) Causes a JavaScript error

Correct Option: B

The submit handler runs and logs "Submitted", but since e.preventDefault() is not called, the browser then performs its default behaviour of submitting the form (usually causing a page reload or navigation).

Q11. Which event is best for running code once the initial HTML has been parsed?
  • A) load on window
  • B) DOMContentLoaded on document
  • C) ready on document
  • D) parseComplete on window

Correct Option: B

document.addEventListener("DOMContentLoaded", ...) fires when the HTML is fully parsed (but before images / CSS etc. are guaranteed loaded).

window.onload waits for everything to load, and there is no built-in ready event.

Q12. Given event delegation on a list, what does event.currentTarget refer to?
  • A) The element that actually started the event
  • B) The element on which the listener is attached
  • C) Always document
  • D) Always window

Correct Option: B

event.currentTarget is the element whose listener is currently running. event.target is the element that started the event (e.g. clicked child).

Q13. Mini-project logic: what does this do?
const list = document.getElementById("task-list");

list.addEventListener("click", (e) => {
  if (e.target.matches("li")) {
    e.target.classList.toggle("done");
  }
});
  • A) Adds a new <li> item on click
  • B) Deletes the clicked <li> item
  • C) Toggles a CSS class done on clicked <li> items
  • D) Marks all items as done

Correct Option: C

This is event delegation: the listener is attached to #task-list (the parent), but when any li inside is clicked, it toggles the done class on the clicked li element.

Q14. Which mouse event fires when a user clicks and releases a button on an element?
  • A) mousedown only
  • B) mouseup only
  • C) click
  • D) mouseover

Correct Option: C

A full ‚Äúclick‚Äù (press + release) triggers first mousedown, then mouseup, and then click. The click event is fired after the release on the same element.

Q15. Form validation snippet ‚Äî what is missing?
form.addEventListener("submit", (e) => {
  const value = input.value.trim();
  if (!value) {
    alert("Name is required");
  }
});
  • A) Need e.preventDefault() inside the if block
  • B) Need return false; after alert
  • C) Need stopPropagation() at the end
  • D) Nothing, it is correct

Correct Option: A

If the value is empty, we should prevent form submission. The clean way: 

if (!value) {
  e.preventDefault();
  alert("Name is required");
}

Otherwise the alert shows and then the form still submits (page reloads).

Q16. What does event.target.value represent in an input event handler?
  • A) The default value of the input
  • B) The current text present in the input at that moment
  • C) The placeholder text
  • D) The label associated with the input

Correct Option: B

In an input/ change handler, event.target is the element that triggered the event (the input), and event.target.value is its current value at the time of the event.

Q17. Which code correctly prevents a link from navigating away?
<a id="link" href="https://example.com">Go</a>
  • A) link.href = "#";
  • B) link.addEventListener("click", e => e.preventDefault());
  • C) link.onclick = () => false;
  • D) Both B and C are valid in practice

Correct Option: D

In modern code, B is preferred (clear and explicit). But C also works in many browsers since returning false from onclick prevents default behaviour.

Setting href="#" (A) changes the destination ‚Äî not a great solution if you still want a real URL.

Q18. When attaching an event listener with capturing phase, which is correct?
  • A) elem.addEventListener("click", handler, true);
  • B) elem.addEventListener("click", handler, false);
  • C) elem.addEventListener("capture", handler);
  • D) elem.addEventListenerCapture("click", handler);

Correct Option: A

The third parameter of addEventListener (or an options object) controls capturing: 

elem.addEventListener("click", handler, true); // capturing phase

false (default) means bubbling phase.

Q19. Output?
<button id="b">Click</button>

<script>
  const b = document.getElementById("b");

  b.addEventListener("click", function () {
    console.log("Normal:", this.id);
  });

  b.addEventListener("click", () => {
    console.log("Arrow:", this.id);
  });
</script>
  • A) Normal: b and Arrow: b
  • B) Normal: undefined and Arrow: undefined
  • C) Normal: b and Arrow: ""
  • D) Normal: b and Arrow: undefined (or something not "b")

Correct Option: D

In the normal function, this refers to the element the listener is bound to, so its id is "b". In the arrow function, this is not rebound ‚Äî it uses the surrounding lexical this (usually window in loose mode, or undefined in strict modules).

Q20. Which event is best for validating a field only after user finishes editing (focus change)?
  • A) input
  • B) change
  • C) keydown
  • D) keyup

Correct Option: B

change on text inputs fires when the element loses focus and the value has changed. That is suitable when you want to validate only after the user is ‚Äúdone‚Äù editing.

Q21. Mini project: what does this ‚Äúlive preview‚Äù code do?
<input id="title-input" type="text">
<h2 id="preview">Live title</h2>

<script>
  const input   = document.getElementById("title-input");
  const preview = document.getElementById("preview");

  input.addEventListener("input", () => {
    preview.textContent = input.value || "Live title";
  });
</script>
  • A) Changes the heading only when the input loses focus
  • B) Updates the heading text in real-time as the user types
  • C) Clears the heading whenever the user types
  • D) Does nothing because textContent is read-only

Correct Option: B

Using the input event ensures the heading‚Äôs text is updated for every keystroke, providing a live preview of what the user types. If the input is empty, it shows a default string.

Q22. Which pattern is most efficient for handling clicks on many dynamically added items?
  • A) Add a separate click listener to every new button
  • B) Use event delegation on a common parent container
  • C) Continuously remove and re-add listeners on scroll
  • D) Poll the DOM every second for changes

Correct Option: B

Event delegation attaches a single listener on a parent and uses event.target to determine which child was clicked. This scales well for many or dynamically-created elements.

Q23. You want to prevent a submit button from being clicked multiple times quickly. Which approach is best?
  • A) Immediately disable the button in the click handler
  • B) Use alert to stop the user
  • C) Call stopPropagation() on the click event
  • D) Wrap the handler with setInterval

Correct Option: A

A common UX pattern is to disable the button as soon as it‚Äôs clicked: 

btn.addEventListener("click", (e) => {
  btn.disabled = true;
  // run submit logic
});

This prevents duplicate submissions. Event propagation is unrelated here.

Q24. In an event handler, which object represents details like mouse position or key pressed?
  • A) window
  • B) document
  • C) The event object passed to the handler
  • D) The DOM element only

Correct Option: C

The event object (often named e or event) contains all details about the action: mouse coordinates, key pressed, target element, timestamp, and more.

Q25. You have this code. What is the effect?
document.addEventListener("click", (e) => {
  if (e.target.matches(".modal-overlay")) {
    closeModal();
  }
});
  • A) Closes the modal whenever any element is clicked
  • B) Closes the modal only when the overlay (background) is clicked
  • C) Closes the modal only when the close button is clicked
  • D) Does nothing because matches() is invalid here

Correct Option: B

This is event delegation on document. When any click occurs, it checks whether event.target matches the selector .modal-overlay. If yes, it calls closeModal(). So the modal closes only when the overlay area is clicked.
