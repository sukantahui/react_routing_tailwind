Section A — Output & Mini Logic (Components)

Q1. “Like” button logic: 
let liked = false;
let count = 0;

function toggleLike() {
  if (liked) {
    liked = false;
    count--;
  } else {
    liked = true;
    count++;
  }
}

toggleLike();
toggleLike();
toggleLike();
console.log(liked, count);
Answer: true 1
Explanation: Calls: 1st: false → true, count: 0 → 1 2nd: true → false, count: 1 → 0 3rd: false → true, count: 0 → 1.
Q2. Star rating hover: 
const stars = [1,2,3,4,5];
let currentHover = 0;

function onStarHover(value) {
  currentHover = value;
}

onStarHover(3);
onStarHover(5);
console.log(currentHover);
Answer: 5
Explanation: The last hover value wins. UI would typically highlight 5 stars.
Q3. Simple slider: 
let value = 50;

function moveSlider(delta) {
  value += delta;
  if (value < 0) value = 0;
  if (value > 100) value = 100;
}

moveSlider(40);   // drag right
moveSlider(30);   // drag right
console.log(value);
Answer: 100
Explanation: 50 + 40 = 90; 90 + 30 = 120 but capped to 100.
Q4. Toggle dropdown open / close: 
let open = false;

function toggleDropdown() {
  open = !open;
}

toggleDropdown();
toggleDropdown();
toggleDropdown();
console.log(open);
Answer: true
Explanation: false → true → false → true after three toggles.
Q5. Stepper component (Next button): 
let step = 1;

function nextStep() {
  if (step < 3) {
    step++;
  }
}

nextStep();
nextStep();
nextStep();
console.log(step);
Answer: 3
Explanation: Steps: 1 → 2 → 3; the last call keeps it at 3 because of limit.
Q6. Accordion: only one open at a time. 
let openId = null;

function toggleAccordion(id) {
  if (openId === id) {
    openId = null;
  } else {
    openId = id;
  }
}

toggleAccordion("q1");
toggleAccordion("q2");
toggleAccordion("q2");
console.log(openId);
Answer: null
Explanation: 1st: null → "q1" 2nd: "q1" → "q2" 3rd: clicking same id again closes: "q2" → null.
Q7. Tabs: 
let activeTab = "overview";

function switchTab(tab) {
  activeTab = tab;
}

switchTab("faq");
switchTab("syllabus");
console.log(activeTab);
Answer: "syllabus"
Explanation: The last tab clicked becomes active.
Q8. Multi-step form “Back” button: 
let step = 2;

function prevStep() {
  if (step > 1) {
    step--;
  }
}

prevStep();
prevStep();
console.log(step);
Answer: 1
Explanation: 2 → 1 → still 1 because minimum step is 1.
Q9. Cart quantity input: 
let qty = 1;

function setQty(newQty) {
  if (newQty < 1) newQty = 1;
  if (newQty > 10) newQty = 10;
  qty = newQty;
}

setQty(5);
setQty(20);
console.log(qty);
Answer: 10
Explanation: second call tries 20, but capped to 10.
Q10. Toast notification auto-hide: 
let visible = false;

function showToast() {
  visible = true;
}
function hideToast() {
  visible = false;
}

showToast();
hideToast();
console.log(visible);
Answer: false
Explanation: Shown then hidden → final state is not visible.

Section B — MCQs on Real-World Event Patterns

Q11. For a dynamic todo list where new items are added with JavaScript, the most scalable way to handle “Delete” button clicks is:

A. Add a separate click listener to each delete button as you create it.
B. Use one listener on the parent list and check event.target.
C. Attach a listener to window and check all clicks.
D. Attach inline onclick handlers in HTML.

Correct Option: B
Explanation: This is event delegation — parent listens for clicks and handles any current or future child buttons efficiently.
Q12. In a modal component, which is the best UX behaviour?

A. Only close when clicking the “Close” button.
B. Close on “Close” button, clicking outside (overlay), and ESC key.
C. Close only on ESC key.
D. Never close automatically; reload the page instead.

Correct Option: B
Explanation: Most modern UIs support multiple ways to dismiss modals for accessibility and user comfort.
Q13. For a range slider controlling video volume, which events are usually combined?

A. click only
B. input for live change + change for final commit
C. mousemove only
D. submit only

Correct Option: B
Explanation: input updates the volume live; change can be used to save the preference.
Q14. In a multi-step form, pressing Enter inside an input should usually:

A. Submit the whole form immediately, even if fields are empty.
B. Be ignored completely.
C. Move to the next step or next field in a controlled way.
D. Close the browser tab.

Correct Option: C
Explanation: Good UX often intercepts Enter to guide the user through steps instead of suddenly submitting everything.
Q15. For a “Scroll to top” button, which event do you mainly listen to in order to show/hide it?

A. resize on window
B. scroll on window
C. click on document
D. keydown on window

Correct Option: B
Explanation: You check scroll position on window and show/hide the button based on how far down the page the user is.

Section C — Real-World Code Reasoning

Q16. “Like” button with text update: 
let likes = 0;
let liked = false;

function onLikeClick() {
  if (liked) {
    liked = false;
    likes--;
  } else {
    liked = true;
    likes++;
  }
}

onLikeClick();  // user clicks Like
onLikeClick();  // user clicks Unlike
onLikeClick();  // user clicks Like again
console.log("Liked?", liked, "Count:", likes);
Answer: Liked? true Count: 1
Explanation: Same pattern as Q1: Like → Unlike → Like again.
Q17. Event delegation on a list: 
<ul id="course-list">
  <li data-course="js">JavaScript</li>
  <li data-course="py">Python</li>
</ul>
const list = document.getElementById("course-list");

list.addEventListener("click", (event) => {
  const li = event.target.closest("li");
  if (!li) return;
  console.log("Clicked:", li.dataset.course);
});
If the user clicks on the text “Python”, what is logged?

Answer: Clicked: py
Explanation: closest("li") finds the clicked list item, and data-course="py" is read via dataset.course.
Q18. Tabs with data-tab: 
<button class="tab-btn" data-tab="overview">Overview</button>
<button class="tab-btn" data-tab="faq">FAQ</button>
const buttons = document.querySelectorAll(".tab-btn");
let active = "overview";

buttons.forEach(btn => {
  btn.addEventListener("click", () => {
    active = btn.dataset.tab;
  });
});

buttons[1].click();
console.log(active);
Answer: "faq"
Explanation: Programmatic click on second button sets active to its data-tab value.
Q19. Form submit with preventDefault: 
let submitted = false;

const form = document.getElementById("demo-form");

form.addEventListener("submit", (e) => {
  e.preventDefault();
  submitted = true;
});

form.dispatchEvent(new Event("submit", { cancelable: true }));
console.log(submitted);
Assume demo-form exists. What will be logged?

Answer: true
Explanation: Dispatching a submit event calls the handler, which sets submitted = true. Page doesn’t reload because of preventDefault().
Q20. Closing modal by clicking overlay: 
<div id="overlay">
  <div id="modal">Modal</div>
</div>
const overlay = document.getElementById("overlay");
const modal   = document.getElementById("modal");
let isOpen = true;

overlay.addEventListener("click", (event) => {
  if (event.target === overlay) {
    isOpen = false;
  }
});

modal.click();
console.log(isOpen);
Answer: true
Explanation: The condition checks if the overlay itself was clicked. Here, event.target is modal, so isOpen stays true.

Section D — Conceptual / Short Answer (Real UI Focus)

Q21. Question: In a “live search” feature for course names at Coder & AccoTax, which event would you use on the search input and why?

Sample Answer:
I would use the input event on the search box because it fires for every change to the field — typing, deleting, pasting, cutting. That lets me filter the course list in real time as the user types, without waiting for blur or Enter.
Q22. Question: Why is event delegation especially useful for a list where users can keep adding new items dynamically?

Sample Answer:
With event delegation, I put one click listener on the parent element and use event.target (or closest()) to figure out which child was clicked. New items added later automatically work without adding more listeners. This reduces memory usage and simplifies the code.
Q23. Question: In a rating widget, why should you store the current rating in a JavaScript variable instead of only relying on the highlighted stars in the DOM?

Sample Answer:
The DOM shows how the rating looks, but the JS variable is the source of truth for logic: sending the rating to a server, restoring it, doing calculations, etc. If you only rely on CSS/DOM state, it’s harder to read and reuse that value in code.
Q24. Question: In a multi-step enrollment form, why is it better to validate each step on its own “Next” button instead of only validating at the final “Submit”?

Sample Answer:
Step-by-step validation gives immediate feedback and prevents users from going forward with half-filled or invalid data. It improves UX, reduces errors at the end, and makes it easier to highlight problems close to where they occur.
Q25. Question: For a “dark mode” toggle switch that changes the theme of the whole Coder & AccoTax site, which element should receive the main class change and why?

Sample Answer:
Usually I add or remove a dark-mode class on the <body> (or a top-level wrapper). Then CSS uses that class to style everything inside. This keeps JS simple (just toggling one class) and keeps the visual rules inside CSS.
