Step 1 · Mindset

1. Why Error Handling & Debugging Matter

Professional JavaScript is not just about “making it work once”. It’s about handling failures, finding bugs quickly, and writing code that is safe, predictable, and readable.

  • Errors will happen — network failures, bad input, logic mistakes.
  • Good code anticipates errors and responds gracefully.
  • Debugging skills save hours of frustration for you and your team.
This module ties together language features (errors), tools (DevTools), and practices (clean code).
Step 2 · Language Feature

2. try...catch & finally Blocks

JavaScript provides try, catch and finally to handle runtime errors without crashing the entire program.

2.1 Basic try...catch

try {
  // Code that might throw an error
  const result = riskyFunction();
  console.log("Result:", result);
} catch (error) {
  console.error("Something went wrong:", error.message);
}
  • try → contains the code that may throw an error.
  • catch → handles the error object.
  • If no error happens, catch is skipped.

2.2 Adding finally

try {
  openConnection();
  // do work
} catch (error) {
  console.error("Error:", error);
} finally {
  closeConnection(); // runs whether error happened or not
}
Use finally for cleanup: closing connections, hiding loaders, resetting UI state, etc.
Step 3 · Throwing Errors

3. Throwing Custom Errors

Sometimes you want to signal that something is wrong: invalid input, missing data, etc. You can use throw to create your own errors.

3.1 Simple throw

function divide(a, b) {
  if (b === 0) {
    throw new Error("Cannot divide by zero");
  }
  return a / b;
}

try {
  console.log(divide(10, 0));
} catch (error) {
  console.error("Division failed:", error.message);
}

3.2 Custom Error Class

class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
  }
}

function validateStudent(student) {
  if (!student.name) {
    throw new ValidationError("Name is required", "name");
  }
  if (student.marks < 0 || student.marks > 100) {
    throw new ValidationError("Marks must be 0-100", "marks");
  }
}

try {
  validateStudent({ name: "", marks: 120 });
} catch (error) {
  if (error instanceof ValidationError) {
    console.error(`Validation failed on ${error.field}: ${error.message}`);
  } else {
    console.error("Unknown error:", error);
  }
}
Using custom error types helps you distinguish between expected errors (e.g. validation) and unexpected bugs.
Step 4 · Understanding Errors

4. Common Runtime Errors & How to Read Them

Every JavaScript developer must learn to calmly read error messages. They are not enemies; they are clues.

4.1 Typical Error Types

  • ReferenceError: using a variable that is not defined.
  • TypeError: using a value in an invalid way (e.g. calling something that is not a function).
  • SyntaxError: code cannot even be parsed (missing brace, wrong punctuation).
  • RangeError: value not in allowed range (e.g. invalid array length).
  • Error: generic error, often thrown by your own code.

4.2 Examples

ReferenceError 
console.log(total); // total is not declared anywhere
// Uncaught ReferenceError: total is not defined
TypeError 
const x = 5;
x(); // trying to call a number like a function
// Uncaught TypeError: x is not a function
SyntaxError 
// Missing closing brace:
if (x > 5) {
  console.log("Hello");
// Uncaught SyntaxError: Unexpected end of input
Train students to look at:
  1. Error type (ReferenceError, TypeError, ...)
  2. Error message (e.g. “x is not a function”)
  3. File name and line number → open that line in editor/DevTools.
Step 5 · DevTools

5. Debugging with Browser DevTools

The browser DevTools (Chrome, Firefox, Edge) are powerful tools to inspect variables, step through code, and understand what is happening at runtime.

5.1 Console for Quick Checks

  • console.log() — print values.
  • console.error() — highlight errors.
  • console.warn() — warnings.
  • console.table() — nice tables for arrays/objects.
console.log("Current user:", user);
console.table(students);
console.error("Unexpected state:", state);

5.2 Breakpoints

  1. Open DevTools → Sources tab.
  2. Open your JS file.
  3. Click on a line number to set a breakpoint.
  4. Reload the page; execution will pause at that line.
Debugger controls: - Continue ► - Step over ⤵ (run next line, skip inside functions) - Step into ↳ (go inside function calls) - Step out ↰ (finish current function and return)

5.3 Watch Expressions & Call Stack

  • Watch: add variables/expressions to watch how values change at each step.
  • Call stack: see which functions were called to reach current line.
Classroom idea: show the same bug first with console.log, then fix it again using a breakpoint and watch expressions, so students see both approaches.
Step 6 · Best Practices

6. Coding Style, Naming & File Structure

Good style doesn’t make your program “more correct”, but it makes it easier to debug, maintain, and collaborate on — which directly reduces bugs.

6.1 Naming Conventions

  • Use clear, descriptive names: totalMarks, studentCount.
  • Functions → verbs: calculateTotal(), loadStudents().
  • Booleans → yes/no: isLoggedIn, hasAccess.
  • Constants → ALL_CAPS only for truly fixed values (e.g. API_BASE_URL).

6.2 Consistent Formatting

  • Use consistent indentation (2 or 4 spaces).
  • Keep lines reasonably short and readable.
  • Use semicolons consistently or follow a clear style rule.
// Good
function calculateAverage(marks) {
  if (!marks.length) return 0;
  const sum = marks.reduce((acc, m) => acc + m, 0);
  return sum / marks.length;
}

6.3 Basic File Structure for a Small Project

project/
  index.html
  assets/
    styles.css
  js/
    app.js
    api.js       // all API related functions
    utils.js     // small helper functions
    validation.js
  • Group related logic in separate files: API, utilities, DOM handlers, etc.
  • Keep each file focused on one responsibility.

6.4 Defensive Programming

  • Validate inputs (e.g. check if string is empty, number is in range).
  • Use if (!data) throw new Error("Data missing") where appropriate.
  • Handle missing properties safely (with optional chaining, defaults).
function getFirstStudentName(classroom) {
  if (!classroom || !classroom.students || classroom.students.length === 0) {
    throw new Error("No students found");
  }
  return classroom.students[0].name;
}
Combining defensive checks + meaningful error messages makes bugs easier to find and fix.
Step 7 · Summary & Practice

7. Summary & Practice Tasks

7.1 Concept Checklist

  • Using try...catch and finally to handle errors.
  • Throwing custom errors and defining error classes.
  • Recognising common runtime errors: ReferenceError, TypeError, SyntaxError, RangeError.
  • Using DevTools: breakpoints, step over/into, watch, call stack.
  • Using the console effectively for logging and debugging.
  • Writing clean, well-structured, defensive JavaScript.

7.2 Practice Exercises

  1. Take a function that can fail (e.g. division, JSON parsing) and wrap it in try...catch, logging a user-friendly message instead of crashing.
  2. Create a StudentError class extending Error and use it in a student validation function.
  3. Intentionally create a ReferenceError and a TypeError in a small file, run them in the browser, and observe the messages + line numbers.
  4. Use DevTools to set a breakpoint in a loop and inspect a variable as it changes on each iteration.
  5. Refactor a messy function: rename variables to clearer names, extract small helper functions, and format properly.
  6. Design a simple file structure for a mini project (HTML + JS + CSS) following the guidelines in this module.
Suggestion: combine this module with a real bug-fixing lab — give students a broken script and ask them to fix it using DevTools and the techniques from this chapter.

Error Handling Quick Reference

  • try/catch: wrap code that may throw.
  • finally: cleanup that must always run.
  • throw: throw new Error("message").
  • Custom error: class MyError extends Error { ... }
  • Common types: ReferenceError, TypeError, SyntaxError, RangeError.
  • DevTools: use breakpoints instead of 100 logs.
  • Console: log, error, warn, table.

Teacher Notes for Module 14

  • Start by showing 2–3 real error messages and reading them slowly with the class.
  • Demonstrate a bug, then fix it twice: once with logs, once with breakpoints.
  • Encourage students to never ignore red errors in the console.
  • Review one of the previous modules’ code and improve its style and error handling live.
  • End with a small “debugging challenge” where students fix an intentionally broken script.
